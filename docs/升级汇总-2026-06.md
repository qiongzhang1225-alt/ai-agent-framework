## 设计取舍记录

详见各 commit message + `public/docs/yuki自查优化-2026-06.md`。

几个反复掂量过的决定：

1. **`code_indexer.update_file` 用单文件增量而非 refresh()**
   yuki 自查建议 refresh()，但 refresh 实际是"pop cache + 全量重建"。改成单文件更新（`update_file`），200+ 文件的重建变成单文件的索引。

2. **`lint(type_check=True)` 默认关，不写硬约束**
   mypy --strict 会爆几百条噪音；非 strict 也漏检。改成按需触发，避免误报淹没真问题。

3. **`audit_stats` nudge 用软提示而非硬阻塞**
   每 20 次自修改后追加 audit_stats 建议到工具返回 msg 末尾。yuki 看到后**自己决定**要不要跑，不强制 —— 因为有时正在做正经活，不该被自查打断。

4. **`smoke_test` 跑子进程而非当前进程**
   ~500ms 启动成本换"看到磁盘真实状态" —— 因为 yuki 改了文件后她内存里还是旧版本，子进程读最新文件才是真验证。

5. **不做"技能自动推荐"系统**
   yuki 自查时建议根据上下文关键词匹配技能。但根本问题不是"找不到技能"，而是"现有技能跟日常任务不对路 + docstring 第一行写得不显眼"。前者通过 audit + 清理解决；后者通过 docstring 重写解决。加推荐系统是治标。
